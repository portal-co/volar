# Garbling Pipeline

The garbling pipeline turns a boolean circuit (`BIrBlocks`) into a pair of
spec-level functions — an **evaluator** and a **garbler** — expressed as
`IrModule` IR that can be compiled to Rust, TypeScript, or C.

Additionally, a **VOLE proving/verifying** pipeline generates Quicksilver-style
ZK proof functions from the same circuits.  See [vole-weaving.md](vole-weaving.md)
for details.

---

## Overview

```
BIrBlocks (boolean circuit with AND / XOR / NOT gates)
        │
        │  volar-weaver
        ├──────────────────────────────────────────────────┐
        │  weave_evaluator(circuit, name, linkage?)        │  weave_garbler(circuit, name, linkage?)
        ▼                                                  ▼
IrModule (evaluator)                               IrModule (garbler)
  generic: <N: ArraySize, D: Digest>                 generic: <N: ArraySize, D: Digest>
  params: one_wire, tables, inputs                   params: one_wire, inputs
  output: Eval<N>                                    output: ([GarbleTable<N>; A], Garble<N>)
```

Both variants are generic over:
- **N** (`ArraySize`) — the byte-length of wire labels. Concretely 16, 32, etc.
- **D** (`Digest`) — the hash function used to derive AND-gate tables (e.g. SHA-256).

---

## Types

All wire types are defined in `volar-spec` and wrap `Array<u8, N>` (a
`GenericArray<u8, N>`).

| Type | Description | Fields |
|---|---|---|
| `Eval<N>` | An evaluated wire label (one byte-array) | `base: Array<u8, N>` |
| `Garble<N>` | A garbled wire label (base share + random delta) | `base: Array<u8, N>` |
| `GarbleTable<N>` | A garbled AND-gate ciphertext | opaque, N bytes each |

XOR gates are free: `Eval` labels are combined element-wise with `^`; `Garble`
labels are similarly XOR'd (bit-by-bit over the byte array).

AND gates require a pre-computed `GarbleTable<N>`, generated by the garbler and
consumed by the evaluator.

---

## Evaluator

Generated by `weave_evaluator`. One function per circuit:

```rust
fn eval_and<N: ArraySize, D: Digest>(
    one_wire:    &Eval<N>,          // the constant-1 wire label
    and_table_0: &GarbleTable<N>,   // pre-computed table for AND gate 0
    input_0:     &Eval<N>,          // input wire 0
    input_1:     &Eval<N>,          // input wire 1
) -> Eval<N>
{
    let wire_0 = input_0.clone();
    let wire_1 = input_1.clone();
    // AND gate: table lookup
    let wire_2 = wire_0.clone().and_via_table::<D>(&wire_1, and_table_0);
    wire_2
}
```

**Gate mapping:**

| `BIrStmt` | Evaluator expression |
|---|---|
| `Xor(a, b)` | `Eval { base: Array::from_fn(\|j\| a.base[j] ^ b.base[j]) }` |
| `And(a, b)` | `a.clone().and_via_table::<D>(&b, &table_k)` |
| `Not(a)` | `a.clone().xor(&one_wire)` (XOR with the constant-1 label) |

The `Or` gate has no direct evaluator operation — the weaver first expands
`Or(a, b)` to `Not(And(Not(a), Not(b)))` (De Morgan's law), introducing
synthetic SSA variables before weaving begins.

## Garbler

Generated by `weave_garbler`. Also one function, but returns the garbled output
label *and* the table list:

```rust
fn garble_and<N: ArraySize, D: Digest>(
    one_wire: &Garble<N>,
    input_0:  &Garble<N>,
    input_1:  &Garble<N>,
) -> ([GarbleTable<N>; 1], Garble<N>)
{
    // XOR gate: element-wise XOR of base shares
    let wire_0_xor_1 = Garble {
        base: Array::from_fn(|j| input_0.base[j] ^ input_1.base[j])
    };
    // AND gate: generate table
    let (table_0, wire_2) = one_wire.clone().gen_and_table::<D>(&input_0, &input_1);
    ([table_0], wire_2)
}
```

**Gate mapping:**

| `BIrStmt` | Garbler expression |
|---|---|
| `Xor(a, b)` | `Garble { base: Array::from_fn(\|j\| a.base[j] ^ b.base[j]) }` |
| `And(a, b)` | `one_wire.clone().gen_and_table::<D>(&a, &b)` → `(GarbleTable<N>, Garble<N>)` |
| `Not(a)` | `a.clone()` XOR with `one_wire` |

The garbler is blocked at the LIR lowering level because fixed-size arrays
of `GarbleTable<N>` require static array support in LIR (the evaluator
already lowers to C).

## `volar-weaver` internals

The weaver lives in `crates/compiler/volar-weaver/src/lib.rs`.

### Or-gate expansion (`expand_ors`)

Before weaving, the circuit is pre-processed: every `BIrStmt::Or(a, b)` is
replaced by:
```
not_a = Not(a)
not_b = Not(b)
and_ab = And(not_a, not_b)
result = Not(and_ab)   // De Morgan: Or(a,b) = Not(And(Not(a), Not(b)))
```

This keeps the weaver's main loop simple: it only needs to handle `And`, `Xor`,
and `Not`.

### SSA assignment model

The circuit's SSA variables (block parameters + statement results) map directly
to `let wire_i = ...;` bindings in the output `IrModule`. The weaver generates
these in order, reusing the SSA numbering from `BIrBlocks`.

### Optional linkage

`weave_evaluator` and `weave_garbler` accept an `Option<&LinkageSystem>`. When
a linkage system that includes the `volar-spec` module is provided, the output
`IrModule.structs` field is populated with the spec's struct definitions
(`Eval<N>`, `Garble<N>`, `GarbleTable<N>`). The struct registry in
`volar-lir-codegen` reads these to register C struct typedefs during lowering.

## Lowering the evaluator to C

After weaving, the evaluator `IrModule` is generic and needs to be
**monomorphized** before LIR lowering:

```rust
use volar_lir_codegen::mono::{MonoEnv, monomorphize_module};
use volar_lir_codegen::lower_module_with_opts;
use volar_c_backend::CBackend;

// 1. Weave
let module = weave_evaluator(&circuit, "eval_and", Some(&linkage));

// 2. Monomorphize: N=16 bytes, D=SHA-256
let env = MonoEnv::new("sha256").with_len("N", 16);
let mono = monomorphize_module(&module, &env);

// 3. Lower to C
let mut backend = CBackend::new();
lower_module_with_opts(&mono, &mut backend, "sha256");
let c_src = backend.finish();

// 4. Compile and test
// cc -O0 -std=c99 -o eval eval.c garble_stubs.c
```

The crypto methods (`.and_via_table::<D>()`, `.gen_and_table::<D>()`) become
`extern` declarations with the hash suffix appended:

```c
extern Eval and_via_table_sha256(Eval recv, Eval other, GarbleTable table);
```

These are implemented by a stub C file during testing.

## Array operations in the garbling IR

All per-byte array operations in the weaver IR use `IrExpr` constructs that
unroll at LIR-lowering time:

| Weaver construct | `IrExpr` form | LIR treatment |
|---|---|---|
| `a.base[j] ^ b.base[j]` for all `j` | `ArrayGenerate { len: N, index_var: "j", body: XOR(a[j], b[j]) }` | Unrolled into N scalar XOR ops, assembled via `arr_new` |
| Struct construction `Garble { base: ... }` | `IrExpr::StructExpr { kind: Custom("Garble"), fields: [("base", array_expr)] }` | `struct_new(garble_id, &[array_val])` |
| Field access `wire.base` | `IrExpr::Field { base: Var("wire"), field: "base" }` | `struct_get(wire_val, 0)` |
| Element access `wire.base[j]` | `IrExpr::Index { base: Field(...), index: Var("j") }` | `arr_get(field_val, j_val)` |

For typical garble parameters (N = 16 bytes), all array operations fully unroll
into 16 scalar byte operations. The resulting C is verbose but straightforwardly
correct and easily verified.
